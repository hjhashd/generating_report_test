1.5.3.2.2.1.5.2 流程触发与初始化

步骤1：业务系统发起决策请求

在业务系统处理关键业务节点（例如客户提交贷款申请或发起大额转账操作）时，若需获取相应的风险判断，则会触发规则引擎的调用流程。此时，业务系统将利用集成的API客户端，向规则引擎的决策端点发起HTTP(S)请求，通常以POST方法调用类似/api/v1/rule-engine/execute的接口。请求体为 JSON 格式，其中包含了本次业务事件的相关上下文数据，例如用户信息、交易金额、操作时间等关键字段，以便规则引擎基于预置规则进行实时分析与决策。

以下为请求示例：

{

"requestId": "req_123456789", // 唯一请求ID，用于幂等和追踪

"sceneCode": "LOAN_PRE_APPROVAL", // 场景码，用于匹配对应的规则集

"timestamp": "2023-10-25T15:30:00Z",

"data": { // 业务数据

"loanApplication": {

"applicationId": "APP202310250001",

"amount": 150000.00,

"term": 36

},

"applicant": {

"customerId": "CUST1001",

"age": 35,

"creditScore": 720,

"annualIncome": 500000.00

}

}

}

步骤2：规则引擎接收并验证请求

当业务系统的请求抵达后，首先由API网关进行处理。API网关作为系统的统一入口，负责接收所有请求，并执行一系列前置安全与管控措施，包括身份认证（例如校验API Key的合法性）、流量控制（防止系统过载）以及基础参数校验。通过网关验证后，请求被转发至规则执行服务。

规则执行服务作为核心决策单元，将进一步对请求体的业务数据进行严格校验，确保必填字段（如标识场景类型的sceneCode和包含业务数据的data）完整，且数据结构符合预期规范。

在整个流程中，任何环节的验证失败（例如认证未通过、请求频率超限或数据格式错误）都会触发系统的异常处理机制。此时，系统将立即中断后续处理，并向客户端返回相应的4xx HTTP状态码及清晰的错误描述信息，从而快速终止流程，确保资源的高效利用和响应的明确性。
