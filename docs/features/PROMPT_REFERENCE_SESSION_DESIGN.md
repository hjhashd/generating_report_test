# 引用提示词会话 + 多次保存收敛/版本刷新：设计说明

本文描述“引用他人提示词卡片进行对话生成/优化，并反复保存迭代”的整体设计思路与预期效果，便于后续排查与继续演进。本文不展开具体代码细节。

## 1. 目标与约束

### 1.1 目标

1. 用户在广场/公共列表中“引用”他人的提示词卡片，进入 Prompt Studio 开始一段新的会话。
2. 引用场景下：不修改对方原提示词，仅基于其内容作为上下文，生成/优化属于用户自己的提示词。
3. 会话需要可回放：刷新/重新打开会话后仍能看到当时引用的提示词卡片内容。
4. 模型需要“有上下文”：每次对话时模型都能获取到被引用卡片的详情，以便进行准确的优化与处理。
5. 聊天记录保留策略：
   - 未保存前：保留该会话的全部聊天记录；
   - 点击保存后：会话收敛为仅保留一条“保存记录”（其余消息软删除）；
   - 保存后继续聊天：新增消息继续保留；下一次保存时再次收敛为仅保留最新一条“保存记录”。
6. 多次保存版本策略：每次保存都刷新为一个新的提示词版本（而不是覆盖旧版本），同时会话绑定到最新版本。

### 1.2 约束

- 不允许对被引用的提示词卡片进行修改（包括内容覆盖、标题覆盖、删除等）。
- 尽量复用现有的会话/保存流程，仅在必要环节增加"引用能力"和"版本化保存"能力。
- 不引入新的 AI 对话接口：引用能力通过现有会话元数据 + 上下文注入实现。
- **引用后必须先对话才能保存**：用户从广场引用提示词后，必须先与 AI 进行至少一次对话（创建有效会话），然后才能保存提示词。这是为了确保保存的内容经过了用户的实际使用和验证。

## 2. 核心数据概念

### 2.1 会话级字段

- ref_prompt_id（引用来源）
  - 含义：本次会话从哪个提示词卡片引用而来（溯源）。
  - 生命周期：创建会话时写入；后续不应被保存流程覆盖。
  - 用途：标识“这段对话的引用来源”，用于前端展示引用状态、用于后端查找引用快照、用于审计/统计。

- origin_prompt_id（会话绑定提示词）
  - 含义：该会话当前绑定到用户自己的哪一条提示词记录（保存结果）。
  - 生命周期：第一次保存后写入；每次保存（尤其是版本刷新）后更新为最新 prompt_id。
  - 用途：在侧边栏区分“已保存提示词会话”、在重复保存场景中判断是否需要“版本刷新”。

### 2.2 消息级机制：引用卡片快照消息（Prompt Ref Snapshot）

仅有 ref_prompt_id 不能满足“回放一致性”和“模型上下文一致性”：

- 回放一致性：未来被引用卡片可能被作者修改/下架，仅靠 id 无法保证历史可回放。
- 模型上下文：模型每次对话需要卡片内容，不可能只靠 id 让模型自行联想。

因此增加一条“引用卡片快照消息”：

- 在会话创建时，从 ai_prompts 读取被引用卡片的关键字段（title/description/content），写入 ai_chat_messages。
- 该消息具备“可识别标记”，用于前端渲染为引用卡片、用于保存收敛时合并进最终保存记录。

这样，引用卡片内容成为“会话历史的一部分”，刷新/回放天然成立。

## 3. 端到端流程设计

### 3.1 引用入口：从提示词卡片进入 Studio

1. 用户在公共列表点击“引用/基于此对话”。
2. 前端进入 Studio，并携带 promptId（被引用提示词 id）。
3. 首次发送消息时后端创建新会话：
   - 写入 ai_chat_sessions.ref_prompt_id；
   - 生成并写入“引用卡片快照消息”到 ai_chat_messages；
   - 返回 session_id 给前端，后续对话全部落在该会话中。

预期效果：
- 引用会话与原提示词实体隔离；对话/保存不会影响对方提示词。
- 会话元数据可追溯：知道引用来源是谁（ref_prompt_id）。
- 刷新/回放可还原：引用卡片快照已在会话消息中固化。

### 3.2 对话时的上下文注入（让模型“看得到引用卡片”）

对话上下文来源分两类：

1. 窗口内短期对话消息（ai_chat_messages）
2. 长期摘要（ai_chat_context_state.history_content），用于长对话压缩

设计要求：
- “引用卡片快照消息”应能参与对话上下文，从而让模型获得引用卡片内容；
- 但快照消息的“技术标记外壳”不应污染模型输入（避免模型输出异常或被标记干扰）。

因此在上下文组装阶段：
- 若检测到“引用快照标记”，剥离标记，仅保留纯文本内容进入模型上下文。

预期效果：
- 模型实际拥有引用卡片的内容上下文；
- 模型不会看到用于前端渲染/合并的标记，从而保持输出稳定性。

### 3.3 保存：一次保存 = 一次会话收敛点

保存发生在 Prompt Studio 的“保存提示词”动作：

保存处理包含两部分：

1. 保存提示词实体（ai_prompts）
2. 收敛会话消息（ai_chat_messages 软删除）

关键点是：收敛不是只做一次，而是“每次保存都收敛一次”。

#### 3.3.1 收敛规则

当用户点击保存时：
- 选择/生成一条“保留消息”（即保存记录）；
- 将会话中除这条之外的其它消息全部标记为 is_deleted=1；
- 更新会话状态为“已保存（completed）”；
- 将 ai_chat_sessions.origin_prompt_id 指向本次保存后的 prompt_id；
- 将 ai_chat_sessions.final_content 更新为本次保存的内容。

预期效果：
- 保存后该会话在历史里看起来只有“一条最终记录”，符合“保存即收敛”的诉求。

#### 3.3.2 为什么“保留消息”要包含引用快照

保存后会话只剩一条消息，如果这条消息不包含引用快照，那么引用信息会随其它消息一起被软删除。

因此保存时对“保留消息内容”做合并：
- 引用卡片快照（如果存在） + 本次最终保存内容（final_content）

预期效果：
- 保存后虽然只剩一条消息，但仍可同时看到“引用来源的内容”和“最终保存结果”。

### 3.4 保存后继续聊天：增量记录在下一次保存前全量保留

保存后会话已处于“已保存”状态，但用户可能继续对话。

设计策略：
- 新产生的消息照常写入 ai_chat_messages（默认 is_deleted=0）。
- 因为上一次保存已经收敛过，所以当前会话“从上一次保存后开始”重新累积增量对话。
- 当用户再次点击保存时，再次执行收敛：只保留这次保存记录，其余软删除。

预期效果：
- 满足“第一次已经保存了，但只要又对话产生新记录，在没有新的保存之前照存不误”。
- 满足“直到新的保存发生，再继续只留一条记录”。

## 4. 多次保存的版本刷新策略（Prompt 版本化）

### 4.1 需求解释

用户希望“每次保存后提示词刷新，成为一个新的提示词”，即：
- 不覆盖旧提示词内容；
- 每次保存产生一个新的 prompt_id；
- 会话绑定到新 prompt_id。

### 4.2 触发条件（避免误分叉）

在“更新提示词（带 prompt_id）”的保存请求中，判断是否需要版本刷新：

- 会话当前 status 为“已保存”
- 且会话当前 origin_prompt_id == 本次请求的 prompt_id
- 且 finalize_session=true（表示这是一次真正的收敛保存）

满足时：创建新提示词版本（新记录），返回新的 prompt_id，并在 finalize_session 中把 origin_prompt_id 更新为新版本。

不满足时：沿用原行为（直接更新已有提示词或新建提示词）。

### 4.3 溯源字段（可选增强）

若 ai_prompts 表具备：
- origin_prompt_id：指向最初祖先版本
- parent_prompt_id：指向直接父版本

则在“版本刷新”创建新版本时：
- origin_prompt_id 继承祖先；
- parent_prompt_id 指向上一版本；

预期效果：
- 后续可扩展“版本树/历史对比/回滚”等能力。

## 5. 前端回放与展示策略

### 5.1 历史加载

前端从会话消息接口获取历史消息后：
- 识别“引用快照标记消息”；
- 将其映射为 UI 的“prompt-ref”类型消息，并用卡片组件渲染；
- 对于普通消息保持原样渲染。

预期效果：
- 刷新页面、重新进入会话时，“引用卡片”不会消失；
- 引用卡片展示内容来自会话快照，而不是依赖路由参数或临时状态。

### 5.2 保存后的展示

保存后会话只剩一条"保存记录"，该记录内容已经包含：
- 引用快照（如果有）
- 本次最终保存内容

因此即使 UI 历史只有一条消息，也能满足"打开能一并看到记录"的诉求。

### 5.3 已保存提示词的特殊标记（实现效果）

为了让用户明确知道哪条消息是"已保存的提示词"，我们实现了特殊标记功能：

**实现机制**：
- 后端在获取会话消息时，同时返回会话元数据（包含 `final_content` 字段）
- 前端在加载历史消息时，将每条消息的内容与 `final_content` 进行比对
- 如果匹配，则将该消息标记为 `type: 'saved'`

**UI 效果**：
- 匹配的消息上方显示绿色渐变背景的"已保存的提示词"标签
- 标签带有文档图标，视觉上清晰可辨
- 即使用户后续进行了新的对话（未保存），也能清楚看到哪条是真正的提示词内容

**设计意图**：
- 简化用户认知：不再显示"引用内容"区块，只显示新会话内容
- 明确标识：通过特殊标记让用户一眼识别已保存的提示词
- 不影响其他功能：点击他人卡片进来时，看到的是该卡片的内容，不受此标记影响

## 6. 保存前置条件检查

### 6.1 业务规则

用户从广场引用提示词后，**必须先进行对话才能保存**。这是为了确保：
1. 用户真正使用了引用的提示词
2. 保存的内容经过了实际验证
3. 避免用户直接复制他人提示词而不做任何修改或使用

### 6.2 技术实现

**判断依据**：
- 会话 ID (`sessionId`) 是判断能否保存的唯一标准
- 引用提示词后首次进入 Studio 时，`sessionId` 为 `null`
- 首次发送消息后，后端创建会话并返回 `session_id`，此时才能保存

**前端交互**：
- 在保存模态框 (`SavePromptModal.vue`) 中检查 `sessionId`
- 如果 `sessionId` 为空：
  - 显示黄色警告提示："无法保存提示词，您刚刚引用了提示词，但尚未进行任何对话或测试。请先与AI进行对话，然后再保存提示词。"
  - 禁用"确认保存"按钮
  - 按钮悬停显示提示："请先进行对话后再保存"
- 如果用户点击保存，弹出 Toast 提示："请先进行对话后再保存提示词"

### 6.3 用户体验流程

```
引用提示词 → 进入 Studio → 显示引用横幅 → 用户输入并发送消息 → 创建会话 → 可以保存
```

## 10. 侧边栏交互优化 + 临时会话功能

### 10.1 需求背景

在之前的实现中，用户通过侧边栏点击提示词卡片进入 Prompt Studio 时：
- 侧边栏仅显示会话列表，没有提示词列表
- 当用户点击自己的提示词但没有会话记录时，显示"引用内容"不够直观
- 用户无法直观看到当前正在引用/编辑哪个提示词

### 10.2 新增功能

#### 10.2.1 标签目录默认收起

**需求**：侧边栏标签目录默认收起，减少视觉干扰

**实现**：
- `StudioSidebar.vue` 中 `isTagsCollapsed` 默认值设为 `true`
- 用户点击可展开/收起标签目录

#### 10.2.2 标签筛选提示词列表

**需求**：点击标签后显示该标签下的提示词列表，而非会话列表

**实现**：
- 选择标签后，查询该标签下的提示词列表（`getPrompts` API，filter='my'）
- 显示提示词卡片，包含标题和描述预览
- 点击提示词卡片进入 Prompt Studio

#### 10.2.3 提示词点击逻辑优化

**需求**：区分自己的提示词和别人的提示词，提供不同的交互方式

**实现逻辑**：

|------|------|---------|

**关键判断**：

### 10.4 新建对话功能优化（2025年2月）

**问题描述**：早期实现中，点击"新建对话"按钮时仅清理了 UI 状态（`currentMode` 和 `lockedMessageId`），但没有清理业务状态，导致：
- 引用提示词后新建对话，引用 banner 仍然显示
- `referencedPrompt` 状态残留，可能导致保存时混淆来源
- `tempSession` 数据未清理，可能干扰新会话
- URL 参数（`promptId`/`session_id`）残留，刷新后可能错误恢复引用状态

**修复方案**（PromptStudio.vue `handleNewChat`）：
```typescript
const handleNewChat = async () => {
  // 1. 清理 UI 状态
  currentMode.value = 'dialogue'
  lockedMessageId.value = null
  
  // 2. 清理引用状态
  referencedPrompt.value = null
  
  // 3. 清理临时会话
  chatStore.clearTempSession()
  
  // 4. 重置聊天到欢迎页
  chatStore.openDraftSession()
  
  // 5. 清理 URL 参数
  if (route.query.promptId || route.query.session_id) {
    const query = { ...route.query }
    delete query.promptId
    delete query.session_id
    await router.replace({ query })
  }
}
```

**预期效果**：
| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 引用卡片后新建对话 | 引用 banner 仍然显示 | 干净进入新对话，无残留 |
| 连续引用不同卡片 | 状态可能混乱 | 每个引用独立，清晰溯源 |
| 引用后未保存就新建 | 旧 tempSession 可能干扰 | 完全隔离，无干扰 |
| 刷新页面 | URL 残留导致错误引用 | URL 清理干净 |
